# 可转债静态报告重构设计

**日期：** 2026-04-25

## 背景

当前项目的数据抓取和汇总链路已经可用，但报告端存在三个明显问题：

1. `render_markdown_parser.py`、`render_html.py`、`base.html.j2`、`style.css`、`app.js` 之间直接共享大量展示细节，字段含义和文案分散，维护成本高。
2. 页面视觉层级弱，核心信息埋在长表格中，用户难以快速发现低估、低溢价、强赎风险和高弹性标的。
3. 交互只有基础筛选和排序，缺少概览图、重点榜单、细节面板和更顺手的浏览路径。

## 目标

1. 保留现有“Python 生成单文件 HTML 报告”的交付方式。
2. 将报告渲染端拆成更清晰的“解析层 / 视图模型层 / 模板层 / 前端交互层”。
3. 重做 UI，让页面更像一份可探索的研究面板，而不是一张很长的表。
4. 增强交互，包括更好的筛选、排序、图表、详情查看和多视图浏览。
5. 为解析和视图模型补上自动化回归测试，避免后续继续在模板里硬编码行为。

## 非目标

1. 不改动 iFinD 抓取、DuckDB 回填、BS 定价和策略评分逻辑。
2. 不把静态报告改造成单独部署的前后端应用。
3. 不为所有数据字段设计复杂配置系统，先聚焦报告阅读体验和代码可维护性。

## 方案比较

### 方案 A：继续修补现有模板

优点是改动最小，最快见效。缺点是现有结构已经把解析、模板和前端逻辑缠在一起，再继续修补只会把复杂度继续堆高。

### 方案 B：改成独立前端应用

优点是体验上限最高，可以做完整状态管理和组件化。缺点是会引入新的打包、部署和数据传输复杂度，不适合当前“本地脚本直接产出 HTML”的使用方式。

### 方案 C：保留静态 HTML 产物，但重建渲染架构

这是本次采用的方案。它兼顾现有流水线稳定性和前端体验提升空间，能在不改变日常使用方式的前提下，显著改善视觉表现、交互和代码结构。

## 设计概要

### 1. 视图模型前置

新增一个 Python 视图模型模块，把 Markdown 解析结果整理成模板和前端都容易消费的结构：

- 页面级摘要：标题、日期、亮点摘要、关键 KPI。
- 探索面板数据：散点图点位、重点榜单、主题分类统计、主题分组摘要。
- 列表视图数据：统一字段名、数值字段、状态标签、搜索文本、卡片摘要。
- 详情面板数据：单只转债的业务描述、主题、策略命中、风险状态、时序数据。

这样模板和 JS 只负责展示和交互，不再现场推导业务逻辑。

### 2. 页面结构重做

新页面分为五个区域：

1. Hero 头部：标题、日期、核心结论、快速统计。
2. 市场雷达区：KPI 卡片、价格/溢价/相对价值散点图、主题热度摘要。
3. 策略观察区：双低、分域双低、低估榜单卡片化呈现。
4. 探索工作台：搜索、主题筛选、风险筛选、视图切换、结果统计。
5. 标的浏览区：桌面端支持卡片视图与数据表视图切换，移动端优先卡片视图；点击任意标的打开详情抽屉。

### 3. 交互策略

前端交互全部基于嵌入式 JSON 完成，不依赖额外接口：

- 全文搜索：转债名、代码、正股、行业、题材、业务描述。
- 快捷筛选：低溢价、低估、高 Delta、强赎触发、价格分层。
- 视图切换：主题分组表格 / 可浏览卡片墙。
- 图表联动：点击散点高亮对应标的并打开详情。
- 详情抽屉：聚合业务描述、状态标签、相对价值、Delta、时序迷你图。
- 导出和复制：只针对当前过滤结果导出。

### 4. 视觉语言

风格方向从“普通深色后台表格”调整为“浅底研究面板 + 编辑化周报”：

- 明确采用浅色背景，不使用大面积黑色或深灰底。
- 主体基底偏米白或浅雾灰，配合更克制的边框、分区卡片和少量高饱和强调色。
- 使用更有个性的中英混排字体组合和更明确的色彩层级。
- 用浅渐变、纸面纹理感、细网格和轻阴影建立背景氛围，但保持专业。
- 用标签、胶囊、编号、榜单卡和细节抽屉强化信息层级。
- 图表和重点数值统一强调色，不再靠零散的红绿文字承担全部视觉表达。

### 5. 测试与验证

新增至少两类测试：

1. Markdown 解析测试：确保标题、摘要、策略、卡片和附录能稳定解析。
2. 视图模型测试：确保 KPI、榜单、过滤数据和详情字段生成正确。

最终验证包括：

- `python -m unittest discover -s tests -v`
- `python -m py_compile ...`
- 重新生成一份 HTML 报告
- 浏览器手工检查主要交互

## 风险与控制

1. 模板和 JS 同时大改，最容易出现字段不一致。
   控制方式：把字段规范收敛到视图模型模块，并用测试锁定关键字段。
2. 静态单文件 HTML 体积会继续增长。
   控制方式：只保留必要的嵌入数据，避免重复序列化。
3. 交互增强后可能影响移动端可读性。
   控制方式：移动端优先展示卡片和抽屉，不强行保留完整宽表。

## 成功标准

1. 报告仍可由现有脚本链路直接生成。
2. 页面能快速回答“今天市场里哪些券值得看、为什么值得看、风险在哪里”。
3. 代码不再把展示逻辑分散在模板、JS 和解析器的多个角落。
4. 自动化测试能覆盖解析和视图模型的关键路径。
